## Problem Statement

Enterprises need AI agents that connect to their company data, run governed (auditable, permissioned, safe), and don't require a $500K consulting engagement to deploy. Today, every enterprise agent deployment is bespoke — built by consultants who leave behind unmaintainable systems with governance bolted on as an afterthought.

Koi is an open-source, self-evolving agent engine where everything is file-based and auditable. Nexus is its virtual file system that seamlessly connects personal agents to company data and groups context together — making it possible for non-technical teams to run governed agents without an engineering department.

## Demand Evidence

- **Founder's direct experience:** Has been in the room during enterprise consulting engagements for AI agents. Has seen firsthand that the governance gap (permissions, audit, safety) is the critical failure point.
- **Money is being spent:** Enterprises are already paying consulting/services firms to build bespoke agent systems. The demand exists — the delivery model is broken.
- **Governance gap is real:** Current agent frameworks treat permissions, audit trails, and safety boundaries as afterthoughts. For enterprises, these are table stakes that block adoption.
- **Gap in evidence:** No named external customer or waitlist yet. The founder is currently their own target user.

## Status Quo

Enterprises wanting to deploy AI agents today have three bad options:
1. **Hire consultants** ($500K+ engagements) who build bespoke systems that die when the consultants leave
2. **Use closed platforms** (OpenAI Assistants, vendor lock-in) that can't be audited, customized, or self-hosted
3. **Do nothing** and wait — which is what most non-technical teams are doing

The consulting model is the most common path for firms with budget. The result: expensive, non-reusable, ungoverned agent systems with no audit trail and no ability to evolve after deployment.

## Target User & Narrowest Wedge

**Target user:** Non-technical team at a small hedge fund (5-50 people). No engineering department. Needs AI agents wired into their business data (market research, email, shared drives, internal docs). Cares deeply about audit trails and knowing what the agent did and why.

**Narrowest wedge:** Deploy Koi + Nexus manually for 3-5 small hedge funds as a concierge service. Wire their data sources into Nexus. Configure agent skills for their specific workflows. Log everything. Learn what "governed agents for non-technical teams" actually means in practice.

## Constraints

- Pre-product: no external users yet
- Founder is a solo technical builder
- Koi's architecture is sophisticated (4-layer system, 7 contracts, ECS composition) — the challenge is making this accessible to non-technical users
- Self-hosted deployment model is a requirement
- Everything must be file-based and auditable

## Premises

1. Core value = open-source, self-evolving agent system where everything is file-based and auditable. Not just a framework for devs — a product non-technical teams can use.
2. Self-hosted deployment.
3. Nexus is a virtual file system connecting personal agents to company data seamlessly — this is what makes Koi defensible vs. every other agent framework.
4. First customers are non-technical teams at small firms (hedge funds, boutique shops) who need agents wired into their data but have no engineering team.

## Approaches Considered

### Approach A: Hedge Fund in a Box

Ship a turnkey, vertical-specific product for small hedge funds. Pre-built agent skills for their workflows (research, summarization, compliance). Nexus pre-wired to common data sources. Single Docker container, self-hosted.

- Effort: M | Risk: Med
- Pros: Fastest to revenue. Narrow = learnable. Non-technical teams can actually use it.
- Cons: Vertical lock-in early. May build the wrong vertical. Hard to generalize later.

### Approach B: Open Engine + Enterprise Edition

Continue building Koi as the open-source extensible engine. Open-source the core, charge for enterprise features (SSO, advanced audit, multi-tenant Nexus). Let technical integrators deploy for any vertical.

- Effort: L | Risk: High
- Pros: Maximum optionality. Community compounds. Classic open-core model.
- Cons: Slow to revenue. Non-technical teams can't use it without an integrator.

### Approach C: Concierge First (Chosen)

Don't ship software yet. Find 3-5 small hedge funds. Deploy Koi manually for each one — be the consultant, but using Koi as the platform. Learn what they actually need. Then productize what worked.

- Effort: S | Risk: Low
- Pros: Maximum learning. Real demand evidence. Revenue from day 1.
- Cons: Doesn't scale. Feels like consulting. Founder time = bottleneck.

## Recommended Approach

**Approach C: Concierge First**, graduating to Approach A once patterns emerge.

The logic: You have direct domain experience but zero external users. The fastest way to validate that non-technical teams at small hedge funds will actually use governed agents is to deploy Koi for them manually and watch what happens. Every manual deployment teaches you:
- Which data sources matter most (Nexus integrations to prioritize)
- Which agent skills get used vs. ignored
- What "auditable" means to a compliance officer vs. a portfolio manager
- Where non-technical users get stuck (the UX gaps)
- Whether anyone would panic if Koi disappeared (demand reality)

After 3-5 concierge deployments, you'll have the evidence to build Approach A (the turnkey product) with confidence — and you'll have paying customers who helped design it.

## Open Questions

1. **How do you find 3-5 small hedge funds?** Do you have a network in finance? Former colleagues? Cold outreach? This is the critical path item.
2. **What does "non-technical" really mean?** Can they use a TUI? Do they need a web UI? A Slack bot? The answer determines a huge amount of product work.
3. **What's the pricing model for concierge?** Monthly retainer? Per-deployment? Free for the first 3 to get the learning?
4. **How much of Koi's current architecture is overkill for the concierge phase?** The 4-layer system and 7 contracts are beautiful — but do you need all of it to deploy for a 20-person hedge fund?
5. **Nexus as virtual file system — what data sources do hedge funds actually need wired in?** Bloomberg? Email? Google Drive? Slack? The answer determines the integration roadmap.

- At least 1 team willing to pay (any amount) for continued service
- Clear pattern emerging across deployments: which skills, which data sources, which governance features
- Enough evidence to write the Approach A spec with confidence
