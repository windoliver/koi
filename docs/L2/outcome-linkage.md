# @koi/outcome-linkage

> Correlation infrastructure connecting agent decisions to downstream business outcomes.

## Why it exists

Koi trajectories track execution-level outcomes (`success` / `failure` / `retry`) but have no
mechanism to link agent decisions to downstream business results. Without outcome linkage, you
have a decision log; with it, you have a decision intelligence platform.

This feature provides Phase 1 correlation infrastructure: types, persistence, and metadata
plumbing so external systems can report outcomes back against specific decisions.

## Layer position

```
L0  @koi/core
    ├── DecisionCorrelationId          (branded ID type)
    ├── OutcomeValence, OutcomeReport  (outcome data types)
    └── OutcomeStore                   (persistence contract)

L2  @koi/event-trace
    └── PERSISTED_TOOL_METADATA_KEYS   (allowlist includes decisionCorrelationId)

L3  @koi/runtime
    ├── outcome-nexus-delegate.ts      (Nexus-backed OutcomeStore)
    ├── outcome-memory-store.ts        (in-memory OutcomeStore for tests/dev)
    └── create-runtime.ts              (wires outcomeStore into RuntimeHandle)
```

**Dependencies**: `@koi/core` (L0), `@koi/fs-nexus` (L2, for NexusTransport).

## Architecture decisions

### 1. Opt-in correlation IDs (not auto-generated)

Correlation IDs are persisted ONLY when callers explicitly set `decisionCorrelationId` in
`ToolResponse.metadata`. Event-trace preserves them through the metadata allowlist but does
NOT auto-generate for every tool call.

**Why**: Auto-generating for all calls violates the event-trace trust boundary rule that
transient correlation IDs stay out of long-lived ATIF metadata. Only decision-bearing steps
(approval, pricing, escalation) should carry persistent linkage.

**Who sets them**: Upstream middleware (approval middleware, governance middleware) or tool
implementations that consider their output a "decision." They generate the ID via
`crypto.randomUUID()` and attach it to the tool response metadata.

### 2. ID allocated before execution

The correlation ID must be set before tool execution so that failures and hook-blocked calls
retain linkage. This is the caller's responsibility — they set the ID on the request, and
event-trace preserves it on both success and failure paths.

### 3. Format-agnostic L0 types

L0 types (`OutcomeReport`, `OutcomeStore`) contain no ATIF-specific concepts. ATIF document
linkage (aggregating correlation IDs into `AtifDocument.extra`) is deferred to Phase 2
where it can be handled in the runtime layer.

### 4. MVP store surface: put + get only

`OutcomeStore` exposes only `put` and `get`. Enumeration (`list`) and deletion (`delete`)
are deferred until a concrete consumer needs them. This keeps the interface minimal.

## API

### L0 types (packages/kernel/core/src/outcome-linkage.ts)

```typescript
type DecisionCorrelationId = Brand<string, "DecisionCorrelationId">
function decisionCorrelationId(id: string): DecisionCorrelationId

type OutcomeValence = "positive" | "negative" | "neutral" | "mixed"

interface OutcomeReport {
  readonly correlationId: DecisionCorrelationId
  readonly outcome: OutcomeValence
  readonly metrics: Readonly<Record<string, number>>
  readonly description: string
  readonly reportedBy: string
  readonly timestamp: number
  readonly metadata?: JsonObject
}

interface OutcomeReportInput {
  readonly correlationId: string
  readonly outcome: OutcomeValence
  readonly metrics: Readonly<Record<string, number>>
  readonly description: string
  readonly reportedBy: string
}

interface OutcomeStore {
  readonly put: (report: OutcomeReport) => Promise<void>
  readonly get: (correlationId: string) => Promise<OutcomeReport | undefined>
}
```

### Nexus outcome delegate (packages/meta/runtime/src/trajectory/)

```typescript
function createNexusOutcomeDelegate(config: NexusOutcomeConfig): OutcomeStore
function createInMemoryOutcomeStore(): OutcomeStore
```

## What it does NOT own

- **Outcome reporting tool** — Phase 2: `createReportOutcomeTool()` wrapping `OutcomeStore.put`
- **ATIF document linkage** — Phase 2: aggregating correlation IDs into `AtifDocument.extra`
- **Cross-session search** — Phase 2: querying outcomes across sessions via `@koi/search-nexus`
- **ACE outcome weighting** — Phase 3: playbook bullets weighted by outcome valence
- **Outcome store enumeration/deletion** — deferred until consumer exists
- **Retention/GC policy** — follow-up issue for `/outcomes/` namespace cleanup
