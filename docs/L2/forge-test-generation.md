# Forge Test Case Auto-Generation (CDGP)

Schema-driven test case generation for forge verification. When a tool is forged
without test cases, the generator produces smoke tests from the tool's `inputSchema`,
ensuring functional correctness is at least probed вҖ” not just "does it crash?"

---

## Why it exists

`ForgeToolInput.testCases` is optional. Tools forged without test cases pass
static analysis, sandbox execution, and adversarial probes вҖ” but **functional
correctness is unverified**. The tool might run without crashing while producing
completely wrong results.

## Architecture

### Layer position

```
L0  @koi/core         в”Җ TestCase, CounterExample (types only)
L2  @koi/forge         в”Җ generateTestCases() + pipeline integration
```

`generateTestCases` is a **pure function**: no side effects, no I/O, no sandbox
calls. It lives in `packages/forge/src/generate-test-cases.ts` and imports only
`TestCase` from `@koi/core`.

### Pipeline integration point

```
verify.ts pipeline:

  Stage 1: Static analysis
  Stage 1.25: Auto-format
  Stage 1.5: Dependency resolve
  Stage 2: Sandbox execution
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  Stage 2.5: Auto-generate tests  в—„в”Җв”Җ NEW (between sandbox and self-test)
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  Stage 3: Self-test + verifiers
  Stage 4: Trust assignment
```

Auto-generation runs **after** sandbox (proving the tool doesn't crash on basic
input) and **before** self-test (so the generated tests are included in the
self-test pass). Agent-provided tests run first, followed by auto-generated ones.

## How it works

### Generation strategies

The generator produces test cases from 5 categories:

| # | Strategy | What it generates | Example |
|---|----------|-------------------|---------|
| 1 | **Minimal valid** | Smallest input satisfying all fields | `{ name: "", age: 0 }` |
| 2 | **Required-only** | Only `required` fields, optional omitted | `{ name: "" }` |
| 3 | **Null variants** | Each optional field set to `null` | `{ name: "", age: null }` |
| 4 | **Boundary values** | Min/max/zero/empty per schema constraint | `""`, `0`, `-1`, `[]` |
| 5 | **Type coercion traps** | Wrong-type values to test input validation | `"1"` for number, `0` for boolean |

### JSON Schema subset supported

The generator handles the common ~90% of JSON Schema:

- **Types**: `string`, `number`, `integer`, `boolean`, `array`, `object`, `null`
- **Object keywords**: `required`, `properties`
- **String constraints**: `minLength`, `maxLength`
- **Number constraints**: `minimum`, `maximum`, `exclusiveMinimum`, `exclusiveMaximum`
- **Array constraints**: `minItems`, `maxItems`, `items`
- **Union keywords**: `oneOf`, `anyOf` (selects first variant)
- **Enum**: `enum` (selects first value)

### Smoke tests (no expectedOutput)

All auto-generated tests are **smoke tests** вҖ” they have `input` but no
`expectedOutput`. They verify the tool doesn't crash, throw, or hang on
schema-derived inputs. This is decision 6A from the design: we can't guess
correct output from schema alone.

### Merging with agent-provided tests

When both exist, tests run in order: **agent-first, then auto-generated** (decision 8A).
This means agent-provided tests with `expectedOutput` run their correctness
assertions first, and auto-generated smoke tests add coverage after.

```typescript
const merged = [...(input.testCases ?? []), ...autoGenerated];
```

### Soft failure

If the generator throws (e.g., on a deeply recursive or malformed schema),
the pipeline **continues with agent-provided tests only** (decision 7A).
Auto-generation is an enhancement, not a gate.

## API reference

### `generateTestCases(schema, config?)`

```typescript
import type { TestCase } from "@koi/core";

interface GenerateTestCasesConfig {
  readonly maxTestCases: number;  // default: 20
}

function generateTestCases(
  schema: Readonly<Record<string, unknown>>,
  config?: GenerateTestCasesConfig,
): readonly TestCase[];
```

**Parameters:**
- `schema` вҖ” JSON Schema describing the tool's input format
- `config.maxTestCases` вҖ” Cap on generated tests (default 20, decision 11C: at least 1, cap at 20)

**Returns:** Readonly array of `TestCase` with `name` prefixed `auto:` and no `expectedOutput`.

**Test name prefixes:**
- `auto:minimal_valid` вҖ” strategy 1
- `auto:required_only` вҖ” strategy 2
- `auto:null_<field>` вҖ” strategy 3
- `auto:boundary_<descriptor>` вҖ” strategy 4
- `auto:coercion_<descriptor>` вҖ” strategy 5

### `SelfTestStageReport`

```typescript
interface SelfTestStageReport extends StageReport {
  readonly stage: "self_test";
  readonly testCasesRun: number;
  readonly agentProvidedCount: number;
  readonly autoGeneratedCount: number;
}
```

Returned by `verifySelfTest` (and surfaced in the `VerificationReport.stages` array).
Tracks how many tests came from the agent vs auto-generation, enabling visibility
into whether a tool was forged with or without explicit test cases.

### `CounterExample`

```typescript
interface CounterExample {
  readonly input: unknown;
  readonly expectedBehavior: string;
  readonly actualBehavior: string;
  readonly recordedAt: number;
}
```

Added to `ToolArtifact.counterexamples` and `ImplementationArtifact.counterexamples`.
Records runtime failures as future regression test inputs. L0 type only вҖ” no
runtime logic in `@koi/core`.

### Configuration

```typescript
// In ForgeConfig.verification:
interface VerificationConfig {
  // ... existing fields ...
  readonly maxAutoTestCases: number;  // default: 20
}
```

Set `maxAutoTestCases: 0` to disable auto-generation entirely.

## Examples

### Tool forged without tests вҖ” auto-generation kicks in

```typescript
const input: ForgeToolInput = {
  kind: "tool",
  name: "addNumbers",
  description: "Adds two numbers",
  inputSchema: {
    type: "object",
    required: ["a", "b"],
    properties: {
      a: { type: "number", minimum: 0 },
      b: { type: "number" },
    },
  },
  implementation: "function run(input) { return input.a + input.b; }",
  // No testCases вҖ” auto-generation will fill the gap
};

// Pipeline auto-generates ~10-15 test cases including:
// - auto:minimal_valid        вҶ’ { a: 0, b: 0 }
// - auto:required_only        вҶ’ { a: 0, b: 0 }  (same as minimal here)
// - auto:boundary_a_minimum   вҶ’ { a: 0, b: 0 }
// - auto:boundary_a_zero      вҶ’ { a: 0, b: 0 }
// - auto:boundary_a_negative  вҶ’ { a: -1, b: 0 }
// - auto:boundary_b_zero      вҶ’ { a: 0, b: 0 }
// - auto:coercion_a_string_for_number вҶ’ { a: "1", b: 0 }
// - auto:coercion_b_string_for_number вҶ’ { a: 0, b: "1" }
// - auto:boundary_empty_object вҶ’ {}
// etc.
```

### Inspecting auto-generation counts in the report

```typescript
const result = await verify(input, context, executor, verifiers, config);
if (result.ok) {
  const selfTest = result.value.stages.find(
    (s) => s.stage === "self_test",
  ) as SelfTestStageReport;

  console.log(selfTest.agentProvidedCount);   // 0
  console.log(selfTest.autoGeneratedCount);   // 12
  console.log(selfTest.testCasesRun);         // 12
}
```

## Edge cases

| Input | Behavior |
|-------|----------|
| Empty schema `{}` | Generates `{ name: "auto:minimal_valid", input: {} }` |
| `type: "object"` with no `properties` | Generates `{ input: {} }` |
| Schema with only required fields (no optional) | Skips `required_only` (identical to minimal) |
| `maxAutoTestCases: 0` | Returns empty array, no tests generated |
| Generator throws | Soft failure вҖ” pipeline continues with agent tests only |
| Non-tool kinds (skill, agent) | Auto-generation is skipped entirely |

## Design decisions

All 16 design decisions were resolved interactively in issue #260:

| # | Decision | Choice |
|---|----------|--------|
| 1 | When to generate | Auto-generate when absent (non-breaking) |
| 2 | Generator location | Inside `@koi/forge` |
| 3 | CounterExample placement | `ToolArtifact` + `ImplementationArtifact` only |
| 4 | Schema subset | Common JSON Schema (~90% coverage) |
| 5 | Pipeline position | Between sandbox and self-test |
| 6 | Test type | Smoke tests (no expectedOutput) |
| 7 | Error handling | Soft failure on generator errors |
| 8 | Merge order | Agent-first, then auto-generated |
| 9 | Test coverage | Both schema types AND generation strategies |
| 10 | Integration tests | In existing verify.test.ts |
| 11 | Bounds | At least 1 test, cap at 20 |
| 12 | Snapshots | Update in same PR |
| 13 | Timeout | Rely on pipeline timeout |
| 14 | Failure mode | failFast + pipeline timeout |
| 15 | Boundary values | Conservative schema-aware |
| 16 | Reporting | SelfTestStageReport with test counts |
